# Foundation Architecture

## Architecture decision

Coder should use its own deterministic workflow runtime as the product kernel.
Other agent systems should be adapters, references, or capability providers.

Recommended foundation:

```text
UI / App Shell
  -> Workflow Library and Template Layer
  -> Coder Workflow Runtime
  -> Context Manager
  -> Agent Executor Adapters
  -> Capability Registry
  -> Memory / RAG Stores
  -> Safety, Approval, Audit, Rollback
```

The key decision is that workflows are product data, not framework-specific
code. A workflow saved by Coder should not become invalid because one external
agent framework changes its API.

## Why not make an external framework the core

### LangGraph

LangGraph is the closest technical reference because it focuses on stateful,
durable, human-in-the-loop agent workflows. However, adopting it as the
product's only core would couple Coder's UI and workflow JSON to LangGraph's
execution semantics.

Decision: use it as a reference and possible future runtime adapter. Keep the
Coder workflow contract independent.

### AutoGen

AutoGen is strong for multi-agent conversations and team-like collaboration,
but Coder needs deterministic visual edges, explicit approvals, path scopes,
patch preview, and rollback. Conversation frameworks are useful below a node,
not as the whole product contract.

Decision: optional adapter for a future `agent_team` node, not the core.

### CrewAI / CrewAI Flows

CrewAI has useful abstractions for agents, crews, and event-driven flows. It is
still a framework-level execution model, while Coder needs user-owned workflow
JSON and local safety controls.

Decision: reference only for flow UX and role-based agent packaging.

### Flowise / Dify

These are strong product references for visual workflow UX, templates, and
low-code authoring. They should not define Coder's core because Coder's product
requires local project safety, patch gates, token-conscious inter-agent
handoffs, and coding-specific rollback.

Decision: borrow UX concepts, not the core.

### OpenAI Agents SDK

The OpenAI Agents SDK is a strong executor layer because it supports agents,
tools, handoffs, guardrails, tracing, and MCP integration.

Decision: future `AgentExecutor` adapter. Do not let it own the workflow graph.

### MCP

MCP is a capability protocol. It lets tools, resources, and prompts be exposed
in a standard way. It is not a permission model by itself.

Decision: MCP servers are installed capabilities. Coder still enforces scopes,
approval, and audit.

## Stable core abstractions

### WorkflowSpec

The workflow remains the source of truth:

```text
workflow
  id
  metadata
  agents
  nodes
  edges
  budgets
  permissions
  template metadata
```

### Node

Initial node types should remain small and stable:

- `start`
- `agent`
- `tool`
- `mcp_tool`
- `condition`
- `human_gate`
- `end`

Future node types should be added only when the current runtime can persist and
resume them safely:

- `loop`
- `parallel`
- `subworkflow`
- `rag_retrieve`
- `memory_write`
- `patch_review`
- `external_agent`

### AgentSpec

Agent configuration should evolve from prompt storage into an execution
contract:

```text
agent
  id
  display_name
  role
  goal
  instructions
  model_policy
  input_contract
  output_contract
  context_policy
  memory_policy
  tool_policy
  permission_policy
```

### ContextPacket

Every agent call receives a context packet. This packet is generated by Coder,
not manually assembled by the previous agent.

```text
context_packet
  task
  upstream_artifacts
  project_context
  knowledge_context
  memory_context
  constraints
  required_output
  provenance
  token_estimate
```

### Artifact

Agents produce artifacts, not loose prose.

Examples:

- `plan_artifact`
- `implementation_artifact`
- `patch_artifact`
- `test_artifact`
- `review_artifact`
- `workflow_design_artifact`

Artifacts should have schemas that can be validated, summarized, and routed.

### Capability

Tools, MCP servers, skills, and future GitHub agent packs should normalize into
one product abstraction:

```text
capability
  id
  type
  display_name
  description
  input_schema
  output_schema
  permissions
  install_source
  runtime_adapter
  risk_level
```

## Recommended execution flow

```text
load workflow
  -> validate graph
  -> create or resume run state
  -> select node
  -> build ContextPacket if node is agent
  -> call executor or tool adapter
  -> validate output artifact
  -> write compact state
  -> emit event
  -> evaluate edges
  -> pause, continue, finish, block, or fail
```

## Reliability requirements

The foundation is not stable until these are true:

- blocked runs can resume after app restart;
- every agent input and output is inspectable;
- every tool call has permission metadata;
- every file mutation has preview, approval, snapshot, and rollback;
- every run has a persisted event log;
- every context packet has provenance;
- token estimates are visible before and after agent calls;
- workflow validation catches broken edges, missing agents, missing tools, and
  invalid output contracts before a run starts.

## Better architecture than free-form agent teams

Free-form multi-agent chat is flexible but weak for reliability. The better
architecture for Coder is:

```text
manual/template workflow graph
  + deterministic runtime
  + structured artifacts
  + centralized context manager
  + explicit capabilities
  + local safety gates
```

This gives users the freedom to build workflows while preserving predictable
execution, inspectability, token control, and rollback.
